# 架构设计

## 整体架构

Kimi Code CLI 采用模块化架构设计，主要分为以下几个层次：

```
┌─────────────────────────────────────────────────────────────────┐
│                        UI 层                                   │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐       │
│  │  Shell   │  │  Print   │  │   ACP    │  │   Wire   │       │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘  └────┬─────┘       │
└───────┼─────────────┼─────────────┼─────────────┼─────────────┘
        │             │             │             │
┌───────▼─────────────▼─────────────▼─────────────▼─────────────┐
│                        Wire 层                                 │
│              (消息传输与事件分发)                               │
└───────┬────────────────────────────────────────────────────────┘
        │
┌───────▼────────────────────────────────────────────────────────┐
│                      KimiSoul (核心循环)                       │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Context → LLM → Tool Call → Context → ... (循环)      │   │
│  └─────────────────────────────────────────────────────────┘   │
└───────┬────────────────────────────────────────────────────────┘
        │
┌───────▼────────────────────────────────────────────────────────┐
│                      Runtime (运行时)                          │
│  ┌─────────┐  ┌─────────┐  ┌───────────┐  ┌─────────────┐     │
│  │ Config  │  │ Session │  │  Approval │  │   LLM      │     │
│  └─────────┘  └─────────┘  └───────────┘  └─────────────┘     │
└───────┬────────────────────────────────────────────────────────┘
        │
┌───────▼────────────────────────────────────────────────────────┐
│                      Toolset (工具集)                          │
│  ┌───────────┐  ┌───────────┐  ┌───────────┐                  │
│  │  Built-in │  │   MCP    │  │  Plugins  │                  │
│  │   Tools   │  │  Tools    │  │          │                  │
│  └───────────┘  └───────────┘  └───────────┘                  │
└────────────────────────────────────────────────────────────────┘
```

## 核心组件

### 1. CLI 入口

**位置**: `src/kimi_cli/cli/__init__.py`

CLI 入口使用 Typer 框架解析命令行参数，然后路由到 `KimiCLI` 进行实际处理。

**主要职责**:
- 解析命令行参数（UI 模式、Agent 规格、配置、MCP）
- 创建或恢复会话
- 初始化运行时环境
- 启动相应的 UI 模式

### 2. KimiCLI

**位置**: `src/kimi_cli/app.py`

`KimiCLI` 是应用的核心入口类，负责：
- 加载配置
- 创建运行时
- 加载 Agent 规格
- 恢复上下文
- 提供多种运行模式（shell、print、acp、wire）

**关键方法**:
- `create()` - 创建 KimiCLI 实例
- `run()` - 运行核心循环
- `run_shell()` - 启动 shell UI
- `run_print()` - 启动打印模式
- `run_acp()` - 启动 ACP 服务器

### 3. Runtime

**位置**: `src/kimi_cli/soul/agent.py`

Runtime 是会话级别的运行时上下文，包含：
- 配置（Config）
- 会话（Session）
- LLM 实例
- OAuth 管理器
- 审批系统
- 钩子引擎
- 通知系统
- 后台任务管理器

### 4. KimiSoul

**位置**: `src/kimi_cli/soul/kimisoul.py`

KimiSoul 是核心代理循环的实现，负责：
- 接受用户输入
- 处理斜杠命令
- 调用 LLM 生成响应
- 执行工具调用
- 管理对话上下文
- 执行上下文压缩

**核心循环流程**:
1. 接收用户输入
2. 追加到上下文
3. 调用 LLM
4. 解析 LLM 响应
5. 执行工具调用（如需）
6. 循环直到生成最终回复

### 5. Context

**位置**: `src/kimi_cli/soul/context.py`

Context 管理对话历史和检查点，支持：
- 消息的追加和持久化
- 系统提示词的管理
- 上下文压缩（Compaction）
- DMail 检查点回复

### 6. Toolset

**位置**: `src/kimi_cli/soul/toolset.py`

Toolset 负责：
- 按导入路径加载工具
- 注入依赖
- 执行工具调用
- 桥接 MCP 工具

### 7. Wire

**位置**: `src/kimi_cli/wire/`

Wire 是 UI 和核心运行时之间的事件总线，支持：
- 消息的流式传输
- 多种消息类型（文本、工具调用、审批请求等）
- 跨进程通信

## 数据流

### 用户输入流程

```
用户输入 → CLI → KimiCLI.create() → Runtime → KimiSoul.run()
                                                          │
                    ┌─────────────────────────────────────┘
                    ▼
              LLM 调用 → 工具调用 → 结果处理 → 回复生成
                    │
                    ▼
              Wire → UI 输出
```

### 配置加载流程

```
配置来源优先级:
1. 命令行参数 (--config, --model, --thinking)
2. 环境变量 (KIMI_MODEL_NAME, KIMI_API_KEY)
3. 配置文件 (~/.kimi/config.toml)
4. 默认值
```

### 会话管理流程

```
会话创建 → 上下文恢复 → 消息追加 → 持久化 → 会话结束
              │                              │
              ▼                              ▼
         历史加载                        状态保存
```

## 关键设计模式

### 1. 依赖注入

Runtime 通过依赖注入模式管理各组件之间的依赖关系，确保组件的松耦合。

### 2. 观察者模式

Wire 系统使用观察者模式实现 UI 和核心运行时的解耦，支持多种 UI 模式同时监听事件。

### 3. 状态机模式

会话状态和审批状态使用状态机模式管理，确保状态转换的正确性。

### 4. 策略模式

工具调用支持多种策略（同步、异步、批量），通过策略模式实现灵活的执行方式。

## 扩展机制

### MCP 工具集成

MCP 工具通过 `fastmcp` 库加载，支持：
- HTTP 传输
- stdio 传输
- OAuth 认证

### Skills 系统

Skills 是可扩展的功能模块，通过 `/skill:<name>` 命令触发，加载对应的 `SKILL.md` 作为用户提示词。

### Hooks 系统

Hooks 允许在特定事件（如会话开始、会话结束、工具调用前后）插入自定义逻辑。

## 安全性设计

### 审批机制

所有工具调用都需要经过审批系统，支持：
- 手动审批模式
- 自动审批模式（yolo 模式）
- AFK 模式（无人值守）

### 权限控制

工具调用会检查文件系统权限，确保只能访问工作目录内的文件。

### 日志审计

所有操作都会记录到日志文件，便于审计和故障排查。
